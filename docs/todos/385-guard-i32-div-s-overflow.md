# TODO: Guard i32.div_s signed overflow

## Problem Description

The `div` instruction emits WebAssembly `i32.div_s` for integer operands. The compiler already rejects division when the divisor is not known to be non-zero, which prevents the ordinary divide-by-zero trap.

However, `i32.div_s` has one additional runtime trap case: signed overflow when dividing the minimum signed 32-bit integer by `-1`.

```8f4e
push -2147483648
push -1
div
```

This can still stop the WebAssembly runtime even though both operands are valid integers and the divisor is non-zero.

## Proposed Solution

Decide how `div` should behave for `-2147483648 / -1`, then implement that behavior explicitly before emitting or replacing the raw `i32.div_s` operation.

Possible approaches:

- Clamp to `2147483647`, matching a saturating arithmetic policy.
- Return `-2147483648`, matching two's-complement wraparound expectations.
- Emit a compiler error only when both operands are compile-time-known constants and document the remaining dynamic risk.
- Add a checked runtime code sequence that branches around the trapping case for dynamic operands.

## Implementation Plan

### Step 1: Pick the language-level behavior

- Define the intended `div` result for `INT_MIN / -1`.
- Document why this behavior differs from raw WebAssembly `i32.div_s`, if applicable.

### Step 2: Update the compiler

- Change `packages/compiler/src/instructionCompilers/div.ts`.
- Keep the existing divisor non-zero validation.
- Add a specific guard for the signed-overflow case.
- Ensure the generated bytecode cannot trap for `INT_MIN / -1` when the chosen behavior requires non-trapping runtime execution.

### Step 3: Add coverage

- Add unit coverage for the instruction compiler bytecode path.
- Add runtime/compiler integration coverage for:
  - `push -2147483648`, `push -1`, `div`
  - ordinary positive division
  - ordinary negative division
  - divisor-zero validation remains unchanged

## Notes

- This is separate from `remainder`: `i32.rem_s` traps on divisor zero, but `INT_MIN % -1` is defined in WebAssembly and returns `0`.
- This is the closest remaining runtime trap behavior to the old trapping `castToInt` conversion issue.
