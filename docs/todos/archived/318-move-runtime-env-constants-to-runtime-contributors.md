# TODO: Move runtime env constants to runtime contributors

## Context

The auto-generated `constants env` block is currently assembled in editor-state by
[auto-env-constants/effect.ts](packages/editor/packages/editor-state/src/features/code-blocks/features/auto-env-constants/effect.ts).

That effect still owns runtime-specific values such as:

- `SAMPLE_RATE`
- `AUDIO_BUFFER_SIZE`

This cuts across the ownership boundary that has been established elsewhere:

- editor-state should own generic parsing and block lifecycle
- each runtime should own the semantics of its own runtime directives and runtime-facing data

The selected runtime already owns sample-rate resolution through runtime-side directive resolvers. The env constants should follow the same boundary.

## Goal

Introduce a runtime contribution API for environment constants so runtime-specific values are generated by the selected runtime, not by editor-state.

## Desired design

Keep a single shared auto-generated `constants env` block, but split responsibility:

- editor-state owns only the generic env-block lifecycle
- the selected runtime contributes its runtime-specific constant lines
- generic non-runtime constants such as binary asset size constants remain outside the runtime contribution API

Suggested runtime registry shape:

```ts
getEnvConstants?: (codeBlocks) => string[];
```

Or equivalent structured return type if metadata becomes useful later.

## Expected behavior

- `WebWorkerLogicRuntime`, `MainThreadLogicRuntime`, `AudioWorkletRuntime`, and `WebWorkerMIDIRuntime` own their own sample-rate-related env constants
- `AudioWorkletRuntime` owns `AUDIO_BUFFER_SIZE`
- editor-state no longer hardcodes runtime-specific env constant values
- switching runtimes updates the generated env block using the selected runtimeâ€™s contribution

## Implementation notes

- Reuse the same runtime-registry ownership model used for runtime directive resolution
- Avoid adding runtime-specific conditionals back into editor-state
- Do not duplicate runtime env-constant data in project config or editor state
- Keep binary asset size constants in the generic env-block pipeline for now

## Acceptance criteria

- [x] Runtime-specific env constants are generated by runtime packages, not editor-state
- [x] editor-state auto-env-constants effect no longer contains sample-rate or audio-buffer-size policy
- [x] the selected runtime contributes env constants through the runtime registry
- [x] env block updates correctly when runtime directives or selected runtime change
